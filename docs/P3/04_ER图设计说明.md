# Camera 约拍服务平台 ER 图设计说明

## 1. 文档定位

本文件说明 Camera 约拍服务平台数据库 ER 图的核心实体、关系基数、字段设计口径与数据约束原则。数据库设计服务于“找人、沟通、预约、交易、交付、评价、仲裁”的完整约拍服务链路，并与核心类图、API 规范文档、建表 SQL 保持统一。

## 2. 设计依据

- P1 需求规格说明书与用户故事
- P2 架构设计文档与 ADR
- 核心类图设计文档
- Camera 平台 API 规范文档
- Camera 核心数据库建表 SQL

数据库设计以项目真实业务闭环为中心，重点支撑服务方展示、需求撮合、双向沟通、正式报价、担保交易、作品交付、照片授权、双向评价和争议处理。

## 3. 设计原则

### 3.1 双向业务优先

系统同时支持需求方发布约拍需求，也支持服务方发布服务橱窗。数据库中使用 `demands` 表承载需求大厅，使用 `service_packages` 表承载服务方橱窗，避免把约拍服务简化为普通商品购买。

### 3.2 报价成单优先

订单由正式报价确认产生。数据库中明确区分沟通、报价和订单三类对象：

- `conversations`：双方沟通关系。
- `quotes`：正式报价单，保存拍摄时间、地点、交付数量、授权范围、合同条款和报价金额。
- `orders`：需求方确认报价后的交易凭证。

该设计保证订单具备明确的来源、内容快照和履约依据。

### 3.3 证据链优先

约拍纠纷常见于风格不符、交付不足、超时、爽约、照片使用授权争议等场景。数据库保留以下证据数据：

- 聊天消息与正式报价记录。
- 订单状态流转日志。
- 交付轮次与交付文件。
- 照片授权记录。
- 评价、评价图片与评价举报。
- 申诉证据与申诉回复。

### 3.4 隐私与权限优先

系统使用 `files` 表统一保存文件元数据，并通过 `visibility` 字段区分公开文件和私有文件。头像、作品集图片、服务橱窗封面可作为公开资源；身份证、学生证、订单交付文件、申诉证据属于私有资源，需要结合文件所有者、订单双方和管理员权限进行访问控制。

### 3.5 状态一致性优先

订单、报价、支付、退款、结算、认证、评价举报、申诉等对象均使用明确状态字段。状态值与 API 规范和核心类图保持一致，避免同一业务含义在不同文档中出现不同枚举名称。

## 4. 核心实体说明

### 4.1 用户与身份

- `users`：用户基础信息，多角色共用一个账号。
- `user_role_bindings`：用户拥有的角色，支持 `CUSTOMER`、`PROVIDER`、`ADMIN`。
- `student_certifications`：学生认证信息。
- `real_name_certifications`：服务方实名认证信息。
- `provider_profiles`：服务方主页，与用户一对零或一。

字段语义约定：

- `user_id` 表示平台用户 ID。
- `provider_user_id` 表示服务方对应的 `users.id`，用于订单、聊天、评价、权限判断等场景。
- `provider_profile_id` 表示服务方主页 `provider_profiles.id`，用于主页、作品集、服务橱窗展示等场景。

### 4.2 文件与隐私

- `files`：统一文件元数据表。

文件本体存放在本地文件系统、MinIO 或对象存储中，MySQL 保存文件标识、存储键、访问 URL、文件类型、业务类型、可见性、大小、所属用户和创建时间。

主要业务类型包括：

- `AVATAR`
- `PORTFOLIO`
- `SERVICE_COVER`
- `ID_CARD`
- `STUDENT_CARD`
- `DEMAND_REFERENCE`
- `DELIVERY`
- `APPEAL_EVIDENCE`
- `REVIEW_IMAGE`

### 4.3 服务方主页、服务橱窗与作品集

- `provider_profiles`：服务方主页资料。
- `service_packages`：服务方发布的服务橱窗。
- `service_package_available_dates`：服务橱窗展示用可约日期。
- `portfolio_works`：作品集作品，区分样片和真实客片。
- `portfolio_images`：作品图片，区分原片和精修。
- `style_tags` 及关联表：支撑风格标签筛选。

真实客片需要具备来源和授权依据。`portfolio_works` 通过 `related_order_id` 与 `photo_authorization_id` 追溯订单来源和照片授权记录。

### 4.4 需求大厅与拍摄企划

- `shooting_plan_templates`：标准化拍摄企划模板。
- `shooting_plans`：用户填写后的具体拍摄企划。
- `demands`：需求方发布的约拍需求。
- `demand_style_tags`：需求与风格标签的关联表。
- `demand_reference_files`：需求参考图文件关联表。
- `demand_responses`：服务方对需求的响应。
- `demand_response_portfolios`：服务方响应需求时附带的作品案例。

拍摄企划用于结构化记录姿势参考、妆造需求、服装风格、修图偏好和安全提示确认信息，减少纯文本沟通带来的遗漏。

### 4.5 沟通与报价

- `conversations`：一对一会话，承载需求响应、服务橱窗、作品集和直接沟通等来源。
- `messages`：消息记录，支持文本、图片、报价卡片和系统通知。
- `quotes`：正式报价单。

`conversations.source_type` 使用以下取值：

- `DEMAND_RESPONSE`
- `SERVICE_PACKAGE`
- `PORTFOLIO`
- `DIRECT`

服务橱窗发起沟通时，系统创建 `conversations` 记录，`source_type = SERVICE_PACKAGE`，`source_id = service_packages.id`，首条咨询内容保存到 `messages` 表。该方式保留“服务橱窗 → 会话 → 报价 → 订单”的完整业务效果。

需求响应被需求方接受后，系统创建 `conversations` 记录，`source_type = DEMAND_RESPONSE`，`source_id = demand_responses.id`。正式订单仍由 `quotes` 确认后生成。

### 4.6 订单、支付、档期与资金状态

- `orders`：订单主表。
- `order_status_logs`：订单状态流转记录。
- `payment_records`：支付、退款和结算相关流水。
- `schedules`：服务方档期。

订单状态统一为：

- `PENDING_PAYMENT`
- `PAID_PENDING_SHOOT`
- `SHOOTING`
- `PENDING_DELIVERY`
- `DELIVERED_PENDING_CONFIRM`
- `APPEALING`
- `REWORK_REQUIRED`
- `COMPLETED`
- `CANCELLED`
- `REFUNDED`

资金相关状态分为三类：

- `escrow_status`：`NOT_PAID`、`HELD`、`RELEASED`、`REFUNDED`。
- `settlement_status`：`NOT_SETTLED`、`SETTLING`、`SETTLED`、`SETTLEMENT_FAILED`。
- `refund_status`：`NONE`、`PENDING_REFUND`、`REFUNDED`、`REFUND_FAILED`。

订单支付成功后进入资金托管状态；需求方确认收货或自动确认后进入结算流程；取消或仲裁退款时更新退款状态与资金托管状态。

### 4.7 交付与照片授权

- `deliveries`：交付轮次，支持返修后的多轮交付。
- `delivery_files`：交付文件，区分原片和精修。
- `photo_authorizations`：照片授权记录。
- `photo_authorization_files`：被授权展示或使用的具体照片。

交付作品属于订单私有成果，不直接等同于公开作品集。服务方将真实客片用于作品集展示时，需要关联已完成订单和照片授权记录。

### 4.8 双向评价、信用与举报

- `reviews`：双向评价。
- `review_images`：评价图片。
- `review_reports`：评价举报。
- `review_report_files`：评价举报证据文件。
- `credit_records`：信用分变动记录。

`reviews.direction` 区分：

- `CUSTOMER_TO_PROVIDER`
- `PROVIDER_TO_CUSTOMER`

同一订单同一评价方向只允许评价一次。信用分通过评价、履约、举报和仲裁结果进行记录和计算。

### 4.9 申诉、仲裁、审核与通知

- `disputes`：订单申诉。
- `dispute_evidence_files`：申诉证据文件。
- `dispute_replies`：申诉回复。
- `dispute_reply_files`：申诉回复附件。
- `audit_records`：后台审核记录。
- `notifications`：站内通知。
- `idempotency_records`：关键写接口幂等记录。

申诉类型覆盖成片质量、数量不足、格式错误、超时交付、需求方爽约、照片未经授权使用、退款争议等约拍场景。

## 5. 关系与基数说明

| 关系 | 基数 | 说明 |
|---|---:|---|
| 用户与角色绑定 | 1:N | 一个用户可拥有多个角色。 |
| 用户与服务方主页 | 1:0..1 | 用户通过服务方认证后拥有服务方主页。 |
| 服务方主页与服务橱窗 | 1:N | 一个服务方可发布多个服务橱窗。 |
| 服务方主页与作品集 | 1:N | 一个服务方可发布多个作品集作品。 |
| 服务橱窗与可约日期 | 1:N | 一个服务橱窗可展示多个可约日期。 |
| 服务橱窗与会话 | 1:N | 多个需求方可从同一服务橱窗发起沟通。 |
| 需求与响应 | 1:N | 一个需求可收到多个服务方响应。 |
| 需求响应与会话 | 1:0..1 | 需求方接受响应后创建会话。 |
| 会话与消息 | 1:N | 一个会话包含多条消息。 |
| 会话与报价 | 1:N | 同一会话可在多次沟通后产生多张报价单。 |
| 报价与订单 | 1:0..1 | 一张已确认报价最多生成一个订单。 |
| 订单与状态日志 | 1:N | 订单状态每次变化记录一条日志。 |
| 订单与支付流水 | 1:N | 一个订单可对应支付、退款、结算等多条流水。 |
| 订单与交付 | 1:0..N | 订单创建后可无交付，履约阶段可存在多轮交付。 |
| 交付与交付文件 | 1:N | 每轮交付包含原片或精修文件。 |
| 订单与照片授权 | 1:N | 一个订单可产生多条照片授权记录。 |
| 照片授权与授权文件 | 1:N | 一条授权记录可覆盖多张照片。 |
| 订单与评价 | 1:0..2 | 双方各自最多评价一次。 |
| 订单与申诉 | 1:N | 一个订单可出现多次不同类型的申诉。 |
| 标签与服务橱窗、需求、作品 | M:N | 通过关联表实现多标签筛选。 |

## 6. 索引设计说明

- 用户与认证：手机号、角色绑定、认证状态字段建立索引，支撑登录、角色判断和后台审核。
- 服务橱窗：服务方、城市、场景、状态、价格字段建立索引，支撑大厅筛选和排序。
- 需求大厅：城市、场景、预算、拍摄日期、状态字段建立索引，支撑服务方浏览需求。
- 会话与消息：会话双方、来源类型、来源 ID、消息时间字段建立索引，支撑会话列表和历史消息查询。
- 报价与订单：报价状态、订单状态、需求方、服务方、支付截止时间字段建立索引，支撑履约流程和定时任务。
- 档期：服务方、开始时间、结束时间、状态字段建立索引，支撑档期冲突检测。
- 交付与授权：订单 ID、交付轮次、授权状态字段建立索引，支撑交付确认和客片追溯。
- 评价与信用：订单、评价方向、被评价用户、信用记录用户字段建立索引，支撑评分计算和重复评价校验。
- 申诉与审核：订单、申诉状态、审核对象类型、审核对象 ID 字段建立索引，支撑后台处理。
- 幂等记录：`idempotency_key` 使用唯一索引，防止关键写接口重复执行。

## 7. 数据安全说明

- 密码只保存哈希值，不保存明文密码。
- 手机号可按业务需要保存脱敏展示字段，普通查询接口不返回完整手机号。
- 身份证号、学生证、证件图片、订单交付作品和申诉证据属于敏感数据。
- 私有文件通过文件权限校验和临时访问地址访问。
- 订单详情、会话消息、交付文件和申诉详情仅允许订单相关双方与管理员查看。
- 客片展示需要具备订单来源与照片授权依据。
- 关键写接口通过幂等记录表避免重复创建订单、重复支付、重复评价或重复申诉。

## 8. 覆盖范围

本 ER 图覆盖以下业务能力：

- 用户注册登录与多角色切换。
- 学生认证与服务方实名认证。
- 服务方主页、服务橱窗、作品集。
- 需求大厅与服务方响应需求。
- 拍摄企划模板与结构化拍摄需求。
- 一对一沟通与正式报价。
- 报价确认生成订单。
- 担保交易、支付、退款、结算。
- 档期临时占用与正式锁定。
- 多轮作品交付与自动确认。
- 照片授权与真实客片追溯。
- 双向评价、评价举报、信用分记录。
- 双方申诉、证据链、后台仲裁。
- 后台审核、站内通知、幂等记录。

数据库设计围绕约拍服务中的信任、预约、履约、出片、授权和争议处理展开，能够支撑项目核心业务闭环和后续功能扩展。
