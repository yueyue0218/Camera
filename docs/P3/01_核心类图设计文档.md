# Camera 约拍服务平台 — 核心类图设计文档

## 一、设计概述

本核心类图基于 P2 阶段确定的 MVC + 分层架构与模块化单体方案，面向 Camera 约拍服务平台的核心业务闭环进行领域建模。系统围绕“找人、沟通、预约、交易、交付、评价、仲裁”的主链路组织类与关系，覆盖 P0 优先级功能，并为后续后端实现、接口实现与数据库设计提供统一语义。

类图采用 Domain 层视角表达核心实体、枚举、领域事件和主要关系。用户认证、文件存储、支付回调、对象存储访问等外部系统能力由 Service 层和基础设施层实现，Domain 层只保留必要的业务属性、状态和领域方法。

### 1.1 设计目标

| 目标 | 说明 |
|---|---|
| 业务闭环完整 | 支撑需求发布、服务橱窗、会话、报价、订单、交付、评价、申诉与仲裁 |
| 模块边界清晰 | 按用户、服务方、作品集、需求、档期、消息报价、订单交易、交付、评价、后台与通知划分 |
| 状态流转集中 | 订单、报价、需求响应、交付、申诉等状态由对应领域方法控制 |
| 文件引用统一 | 头像、作品、证件、交付作品、申诉证据统一通过 StoredFile 关联 |
| 隐私权限可控 | 私有文件不直接公开 URL，由文件服务按业务上下文校验访问权限 |
| 后续实现友好 | 与 API 规范、ER 图和建表 SQL 使用同一组术语、枚举和 ID 语义 |

### 1.3 ID 语义约定

| 名称 | 含义 | 使用位置 |
|---|---|---|
| `userId` | 平台用户统一 ID | 用户资料、登录态、通用权限判断 |
| `providerUserId` | 服务方对应的用户 ID | 订单、报价、会话、评价、档期、仲裁 |
| `providerProfileId` | 服务方主页 ID | 主页展示、作品集、服务橱窗 |
| `customerId` | 需求方对应的用户 ID | 需求、报价、订单、评价、申诉 |
| `fileId` | 文件元数据 ID | 头像、作品、证件、交付作品、申诉证据 |
| `sourceType + sourceId` | 会话或报价来源 | 需求响应、服务橱窗、作品集、直接沟通 |

API 对外返回 `providerId` 时，语义为服务方用户 ID，即内部的 `providerUserId`。

## 二、各模块类设计

### 2.1 用户与身份模块

**User** 是平台账号主体，同一账号可具备 CUSTOMER、PROVIDER、ADMIN 中的一种或多种角色，并通过 `currentRole` 表示当前使用视角。User 只负责用户基础资料、角色切换和信用等级计算，不承担短信验证码、密码校验、Token 签发等认证流程。

**StudentCertification** 记录学生认证材料和审核状态，用于支撑学生身份可信度展示。

**RealNameCertification** 记录服务方实名认证材料和审核状态，用于服务方入驻、信用展示和交易安全保障。

### 2.2 文件资源模块

**StoredFile** 是统一文件元数据类，保存文件归属人、业务类型、可见性、存储键、公开 URL、文件类型和大小。文件本体由对象存储或本地存储保存，Domain 层只引用 `fileId`。

文件可见性分为 PUBLIC 和 PRIVATE。头像、作品集展示图、服务橱窗封面通常为 PUBLIC；身份证、学生证、订单交付作品、申诉证据必须为 PRIVATE。私有文件访问由文件服务结合用户身份、订单关系和管理员权限判断。

### 2.3 服务方主页模块

**ProviderProfile** 表示服务方主页，与 User 一对一。该类保存服务类型、简介、风格标签、服务区域、价格区间、接单状态、评分和已完成订单数。

**ServicePackage** 表示服务方发布的服务橱窗。它不是普通商品，而是约拍服务入口，包含服务标题、封面、场景、城市、地点说明、基础价格、拍摄时长、原片数量、精修数量、交付天数、适用人数、服务内容说明和预约须知。

服务橱窗支持 DRAFT、ONLINE、OFFLINE、PENDING_REVIEW、REJECTED 五类状态，用于表达编辑、上架、下架和后台审核结果。

消费者从服务橱窗发起咨询时，系统直接创建 Conversation，并将 `sourceType` 记为 SERVICE_PACKAGE、`sourceId` 记为服务橱窗 ID。系统不单独建立服务咨询实体。

### 2.4 作品集与照片授权模块

**PortfolioWork** 表示服务方公开展示的作品。作品归属于 ProviderProfile，同时保存 providerUserId 以支持权限判断。作品类型通过 `WorkCategory` 区分 SAMPLE 和 CUSTOMER_WORK。

**WorkImage** 表示作品中的图片项，通过 `fileId` 关联 StoredFile，并通过 `ImageType` 区分 RAW 与 RETOUCHED。

真实客片进入公开作品集前，需要能够追溯来源订单或照片授权依据。**PhotoAuthorization** 表示客户对照片展示或使用范围的授权，**PhotoAuthorizationFile** 记录具体被授权的文件。PortfolioWork 可通过 `relatedOrderId` 和 `photoAuthorizationId` 形成证据链。

### 2.5 需求发布模块

**Demand** 表示需求方发布的约拍需求，包含拍摄场景、风格标签、期望日期、地点、预算范围、参考图、描述、状态和过期时间。

**DemandResponse** 表示服务方对需求的响应。需求方接受响应后，系统创建 Conversation，并将 `sourceType` 记为 DEMAND_RESPONSE、`sourceId` 记为响应 ID。订单仍然必须通过正式报价确认产生。

需求状态用于表达从草稿、开放、匹配、报价、成单到关闭或过期的业务阶段。响应状态用于表达待需求方接受、已接受、已拒绝、已撤回、已报价和已成单。

### 2.6 档期模块

**Schedule** 表示服务方可预约或已占用的时间段。支付前可产生临时锁定，支付成功后进入正式占用。`lockExpireTime` 用于支付超时后的释放判断。

档期状态包括 AVAILABLE、LOCKED、BOOKED。订单支付成功事件会触发档期锁定，订单取消或退款完成后可释放相关档期。

### 2.7 消息与报价模块

**Conversation** 表示一对一沟通会话。会话来源由 `sourceType + sourceId` 表达，支持从需求响应、服务橱窗、作品集或直接沟通进入。

**Message** 表示会话消息，支持文本、图片、报价卡片和系统通知。图片消息通过 `fileId` 关联 StoredFile；报价卡片通过 `referenceType` 和 `referenceId` 关联 Quote。

**Quote** 表示正式报价单，是订单生成前的准合同凭证。报价单包含拍摄时间、地点、服务内容、原片数量、精修数量、交付截止时间、照片使用范围、条款、合同快照、安全提示版本、总金额、过期时间和关联订单 ID。

报价确认后，系统检查档期并创建订单；拒绝或撤回报价不会关闭会话，双方仍可继续沟通并再次报价。

### 2.8 订单与交易模块

**Order** 是交易与履约的核心实体，基于 Quote 创建。订单保存需求方、服务方、报价、会话、需求或服务橱窗来源、订单状态、资金状态、金额、拍摄时间地点、交付期限、照片使用范围、报价快照、安全提示确认和合同条款。

订单状态流转由 Order 的领域方法控制，Controller 禁止直接赋值 status 字段。核心状态链路如下：

```text
PENDING_PAYMENT -> PAID_PENDING_SHOOT -> SHOOTING -> PENDING_DELIVERY -> DELIVERED_PENDING_CONFIRM -> COMPLETED
```

异常分支包括取消、申诉、返修和退款：

```text
PENDING_PAYMENT -> CANCELLED
PAID_PENDING_SHOOT -> CANCELLED / APPEALING
SHOOTING -> APPEALING
PENDING_DELIVERY -> APPEALING
DELIVERED_PENDING_CONFIRM -> APPEALING / REWORK_REQUIRED
APPEALING -> COMPLETED / REWORK_REQUIRED / REFUNDED
REWORK_REQUIRED -> PENDING_DELIVERY / APPEALING
```

**OrderStatusLog** 记录订单状态流转时间线。**PaymentRecord** 记录支付与退款流水。担保交易通过 `EscrowStatus`、`SettlementStatus`、`RefundStatus` 分别表达资金托管、服务方结算和退款处理状态。

### 2.9 作品交付模块

**Delivery** 表示订单交付轮次，支持返修后的多轮交付。订单刚创建、待支付、待拍摄阶段可以没有交付记录。

**DeliveryFile** 表示交付文件，使用 `fileId` 关联 StoredFile，并通过 ImageType 区分原片和精修。交付文件属于订单私有成果，不会自动进入公开作品集。

### 2.10 评价与信用模块

**Review** 表示双向评价，通过 `ReviewDirection` 区分 CUSTOMER_TO_PROVIDER 和 PROVIDER_TO_CUSTOMER。同一订单同一方向只能评价一次。

**CreditRecord** 记录信用分变动来源。评价、爽约、仲裁结果、违规使用照片等事件都可以生成信用记录。

### 2.11 后台审核与仲裁模块

**Dispute** 表示订单申诉与仲裁。申诉可由订单双方发起，类型覆盖风格不符、质量问题、数量不足、格式错误、超时交付、需求方爽约、未经授权使用照片、退款争议和其他问题。

申诉支持申请人证据、被申诉方回复、管理员处理意见、退款金额和处理时间。一个订单可以存在多条申诉记录。

**AuditRecord** 表示后台审核记录，覆盖实名认证、学生认证、服务方主页、服务橱窗、作品集、评价举报和申诉处理。

### 2.12 通知模块

**Notification** 表示站内通知。通知通过 `relatedType + relatedId` 指向订单、会话、需求、评价、申诉或审核对象，支持前端跳转到具体业务页面。

## 三、类间关系总结

### 3.1 关联关系

- User 与 ProviderProfile：一对零或一。
- User 与 StudentCertification / RealNameCertification：一对零或一。
- ProviderProfile 与 ServicePackage：一对多。
- ProviderProfile 与 PortfolioWork：一对多。
- Demand 与 DemandResponse：一对多。
- Conversation 与 Quote：一对多。
- Quote 与 Order：一对零或一。
- Order 与 User：多对一，分别关联需求方和服务方。
- Order 与 Conversation：多对一。
- PortfolioWork 与 PhotoAuthorization：多对一或可为空。

### 3.2 组合关系

- ProviderProfile 组合管理 PortfolioWork、Schedule、ServicePackage。
- Conversation 组合管理 Message。
- Order 组合管理 OrderStatusLog 与 PaymentRecord。
- Delivery 组合管理 DeliveryFile。
- PhotoAuthorization 组合管理 PhotoAuthorizationFile。
- User 组合管理 Notification 与 CreditRecord。

### 3.3 聚合关系

- Order 聚合 Delivery，支持零到多轮交付。
- Order 聚合 Review，最多两条评价记录。
- Order 聚合 Dispute，支持多次不同类型申诉。
- Demand 聚合 DemandResponse。

### 3.4 依赖关系

- Order 发布 OrderPaidEvent、OrderCompletedEvent、OrderCancelledEvent、RefundCreatedEvent。
- Delivery 发布 DeliveryUploadedEvent、DeliveryAutoConfirmedEvent。
- Review 发布 ReviewCreatedEvent。
- Dispute 发布 DisputeOpenedEvent、DisputeResolvedEvent。

## 四、领域事件设计

| 事件名 | 触发时机 | 主要监听方 | 处理逻辑 |
|---|---|---|---|
| OrderPaidEvent | 订单支付成功 | 档期模块、通知模块 | 锁定档期并通知双方 |
| OrderCompletedEvent | 订单完成 | 评价模块、结算模块、通知模块 | 开放评价入口并发起结算 |
| OrderCancelledEvent | 订单取消 | 档期模块、通知模块 | 释放档期并通知对方 |
| DeliveryUploadedEvent | 服务方上传交付作品 | 订单模块、通知模块 | 更新订单为已交付待确认并通知需求方 |
| DeliveryAutoConfirmedEvent | 交付后到达自动确认时间 | 订单模块、评价模块、通知模块 | 自动完成订单并开放评价入口 |
| ReviewCreatedEvent | 用户提交评价 | 信用模块、通知模块 | 计算信用分并通知被评价方 |
| DisputeOpenedEvent | 用户发起申诉 | 通知模块、后台模块 | 通知被申诉方并进入处理流程 |
| DisputeResolvedEvent | 仲裁处理完成 | 订单模块、信用模块、通知模块 | 更新订单、信用记录和双方通知 |
| RefundCreatedEvent | 产生退款处理 | 支付模块、通知模块 | 调用退款流程并通知双方 |

## 五、设计模式应用

### 5.1 状态模式：订单状态机

订单状态通过 OrderStatus 枚举和 Order 领域方法共同管理。每个状态只允许流转到业务规则许可的目标状态，Order 在执行 `pay()`、`cancel()`、`startShooting()`、`markPendingDelivery()`、`confirmDelivery()`、`applyDispute()`、`refund()` 等方法前完成状态校验。

该设计把状态规则集中在领域层，避免 Controller 或 Service 中散落大量状态判断，降低非法状态流转风险。

### 5.2 观察者模式：领域事件

订单支付、订单完成、订单取消、作品交付、评价创建、申诉发起、仲裁完成等关键节点通过领域事件通知其他模块。各模块根据自身职责监听事件，完成档期锁定、消息通知、信用分计算、结算处理等动作。

该设计降低了订单模块与档期、通知、评价、结算等模块之间的直接耦合，符合模块化单体的分层设计目标。

## 七、实现约束

1. 服务橱窗咨询不建立独立咨询类。消费者点击咨询后直接创建 Conversation，并写入首条 Message。
2. 订单必须由 Quote 确认产生，不支持绕过报价直接创建交易订单。
3. 订单状态只能通过 Order 领域方法流转，禁止直接设置 `status` 字段。
4. 私有文件不直接暴露永久公开 URL，必须通过文件服务进行访问校验。
5. 真实客片进入作品集时，必须关联完成订单或照片授权记录。
6. 同一订单同一评价方向只能存在一条 Review。
7. 同一时刻同一会话只能存在一个待确认报价。
8. 支付、确认报价、确认收货、评价、申诉等关键写操作需要配合幂等记录处理。
