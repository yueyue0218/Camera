# Camera 约拍服务平台 P3 详细设计文档

## 1. 文档定位与设计目标
本文件面向 Camera 约拍服务平台 P3 阶段的详细设计交付，整合核心类设计、API 接口设计、数据库设计、设计模式应用、SOLID 原则落实和关键约束规则。文档采用统一术语描述系统，确保前端、后端、数据库和项目文档能够围绕同一套业务模型开展后续实现。

Camera 平台面向校园及年轻用户的约拍服务场景，核心业务链路为“找人、沟通、预约、交易、交付、评价、仲裁”。系统同时支持需求方发布约拍需求，也支持服务方发布服务橱窗，通过会话沟通和正式报价形成订单，并通过担保交易、作品交付、照片授权、评价信用和争议处理形成完整闭环。

## 1.1 设计目标
| **目标**     | **说明**                                                                               |
|--------------|----------------------------------------------------------------------------------------|
| 业务闭环完整 | 覆盖需求发布、服务橱窗、会话、报价、订单、交付、评价与争议处理。                       |
| 术语统一     | 统一使用 CUSTOMER、PROVIDER、ADMIN、providerUserId、providerProfileId、fileId 等语义。 |
| 状态明确     | 订单、报价、需求响应、交付、认证、评价举报和争议处理均具备清晰状态。                   |
| 接口可实现   | API 使用统一响应包装、统一错误码、鉴权规则、分页规范和幂等要求。                       |
| 数据可追溯   | 订单状态日志、报价快照、交付文件、照片授权、评价和争议证据形成证据链。                 |
| 隐私可控     | 证件、学生证、交付作品和争议证据均作为私有文件管理。                                   |

## 2. 总体架构与模块划分
系统采用 MVC + 分层架构 + 模块化单体方案。移动端与后台管理端通过 RESTful API 调用后端服务，后端按 Controller、Application Service、Domain、Repository 和 Infrastructure 层组织代码。数据库采用 MySQL，文件本体可存放在本地文件系统、MinIO 或对象存储中，MySQL 只保存文件元数据。

## 2.1 分层职责
| **层次**            | **职责**                                                               |
|---------------------|------------------------------------------------------------------------|
| Controller          | 接收请求、完成基础参数绑定、调用应用服务并返回统一响应。               |
| Application Service | 编排跨领域流程，例如确认报价、支付回调、自动确认、争议处理和通知发送。 |
| Domain              | 保存领域实体、值对象、枚举、状态流转方法和核心业务规则。               |
| Repository          | 完成实体持久化、查询封装和事务边界内的数据访问。                       |
| Infrastructure      | 对接短信、文件存储、支付、消息推送和定时任务等基础能力。               |

## 3.4 服务橱窗沟通模型
服务橱窗的咨询入口不建立独立业务实体。消费者从服务橱窗进入沟通时，系统直接创建 Conversation，将 sourceType 设置为 SERVICE_PACKAGE，将 sourceId 设置为服务橱窗 ID，并把用户填写的咨询内容保存为首条 Message。后续由服务方在会话中发送正式报价，订单仍由需求方确认报价后产生。

> ServicePackage -\> Conversation(sourceType = SERVICE_PACKAGE) -\> Quote -\> Order

需求大厅路径与服务橱窗路径保持同一成单原则。服务方响应需求后，需求方接受响应，系统创建 Conversation，sourceType 为 DEMAND_RESPONSE，sourceId 为响应 ID。双方继续沟通后由正式报价生成订单。

> Demand -\> DemandResponse -\> Conversation(sourceType = DEMAND_RESPONSE) -\> Quote -\> Order

## 4. API 接口设计
API 采用 RESTful 风格，统一使用 HTTPS、JSON 数据格式、ISO-8601 时间格式和人民币元金额格式。除公开浏览、验证码和登录注册外，其余接口均需要 Authorization: Bearer token。所有响应使用统一包装结构。

## 4.1 统一响应与错误码
> {  
> "code": 200,  
> "message": "success",  
> "data": {}  
> }

失败响应同样使用统一结构，并通过 code、message、data 表达错误类型、错误信息和必要上下文。核心错误码覆盖参数错误、鉴权失败、权限不足、资源不存在、档期冲突、订单状态不允许、报价过期、重复评价、支付金额不一致、合同确认缺失、交付不完整和第三方服务不可用。

## 4.3 幂等要求
关键写接口必须携带 Idempotency-Key 请求头，避免重复点击、网络重试导致重复创建资源或重复推进状态。适用接口包括确认报价、创建支付单、确认收货、提交评价和发起争议。

## 5.3 索引设计
索引围绕登录检索、列表筛选、权限查询、状态推进和时间排序设计。手机号使用哈希字段支撑登录与唯一性校验；服务橱窗按城市、场景、价格、状态建立查询索引；需求按城市、场景、预算、日期和状态建立筛选索引；订单按 customer_id、provider_user_id、status 和创建时间建立列表索引；会话消息按 conversation_id 与创建时间排序；文件按 uploader_id、biz_type、visibility 建立查询索引。

## 6. 设计模式应用
| **设计模式** | **应用位置**                                         | **使用理由**                                                              | **不用该模式的影响**                                   |
|--------------|------------------------------------------------------|---------------------------------------------------------------------------|--------------------------------------------------------|
| 状态模式     | Order、Quote、DemandResponse、Dispute 的状态流转     | 状态转换由领域对象集中校验，避免 Controller 或 Service 直接修改状态字段。 | 状态规则容易散落在多个接口中，出现非法流转和重复判断。 |
| 策略模式     | 支付、退款、文件访问地址生成、评价信用分计算         | 不同支付渠道、存储渠道或信用计算规则可在接口层保持稳定。                  | 更换实现时需要改动业务编排代码，影响范围扩大。         |
| 观察者模式   | 订单支付成功、交付确认、争议开启、退款完成等领域事件 | 主流程完成状态推进，通知、档期、结算等副作用通过事件订阅处理。            | 主流程会包含大量外部调用，耦合度高，失败恢复复杂。     |

## 7. SOLID 原则落实
| **原则** | **落实方式**                                                                                       |
|----------|----------------------------------------------------------------------------------------------------|
| 单一职责 | 实体类只负责自身数据和核心领域规则；跨模块编排由 Application Service 承担。                        |
| 开闭原则 | 订单状态、报价状态、文件存储、支付和信用计算通过枚举分组、接口和策略扩展。                         |
| 里氏替换 | 学生认证和实名认证保持独立实体，共用状态枚举，不强行建立不稳定继承层级。                           |
| 接口隔离 | User 不承担登录注册流程；认证能力由 AuthService、Session API 和验证码接口提供。                    |
| 依赖倒转 | 领域层通过 fileId、PaymentRecord、领域事件表达业务事实，底层存储、支付和短信实现由基础设施层提供。 |

核心类图、API、ER 图和 SQL 均围绕以上原则展开，保证业务规则集中、外部实现可替换、核心状态可追溯。

## 8. 安全、隐私与幂等约束
## 8.1 鉴权与角色
除验证码、登录注册、公开浏览服务方主页、作品集、服务大厅和评价外，其他接口均需要用户登录。CUSTOMER 可发布需求、确认报价、支付订单、确认收货、评价和发起争议；PROVIDER 可维护主页、发布服务橱窗、响应需求、发送报价、维护档期、上传交付作品和回复评价；ADMIN 可处理认证、争议、举报和运营管理。

## 8.2 隐私数据
手机号保存密文、哈希和脱敏展示值；密码仅保存 BCrypt 哈希；身份证号、学生证号和证件图片不在普通接口中返回完整值；证件、学生证、交付作品和争议证据必须作为私有文件保存。私有文件访问需要结合文件所有者、订单关系、会话关系或管理员权限判断。

## 8.3 幂等与证据链
确认报价、创建支付单、确认收货、提交评价和发起争议均需要幂等键。系统保存聊天消息、报价快照、订单状态日志、交付轮次、照片授权、评价和争议证据，确保交易过程可以追溯。

## 9. 一致性口径清单
| **主题**     | **统一口径**                                                                  |
|--------------|-------------------------------------------------------------------------------|
| 服务橱窗沟通 | 不建立独立服务咨询实体；点击咨询直接创建 Conversation 并保存首条 Message。    |
| 会话来源     | 服务橱窗使用 SERVICE_PACKAGE，需求响应使用 DEMAND_RESPONSE。                  |
| 成单原则     | 订单必须由正式报价确认产生，不从服务橱窗直接购买。                            |
| 文件引用     | 业务对象通过 fileId 关联 StoredFile，私有文件通过权限校验后获取临时访问地址。 |
| 真实客片     | CUSTOMER_WORK 需要 relatedOrderId 或 photoAuthorizationId 支撑来源与授权。    |
| 订单状态     | 统一使用 10 个 OrderStatus，APPEALING 和 REWORK_REQUIRED 为争议相关状态。     |
| 资金状态     | escrowStatus、settlementStatus、refundStatus 分别表达托管、结算和退款。       |
| 服务方 ID    | 对外 providerId 表示 providerUserId；主页展示使用 providerProfileId。         |
