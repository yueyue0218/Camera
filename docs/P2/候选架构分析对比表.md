
| **对比维度**     | **纯 MVC 架构**                                    | **纯分层架构**                            | **微服务架构**                              | **MVC + 分层架构**                                 |
| ------------ | ----------------------------------------------- | ------------------------------------ | -------------------------------------- | ---------------------------------------------- |
| **开发复杂度**    | **极低**。建好 Model 和 Controller 直接写逻辑，像写脚本一样快。     | **中等**。前期需要设计接口、划分边界（如 DTO 转换），起步稍慢。 | **极高**。需要处理服务拆分、网关、注册中心、分布式数据库等。       | **中低**。利用成熟框架（如 Spring Boot）的规范，配置简单，兼顾了速度与结构。 |
| **可维护性**     | **低（中后期）**。业务一旦复杂，Controller 会瞬间变成几千行的垃圾堆，极难维护。 | **高**。各司其职，改数据库不影响业务，改业务不影响上层。       | **低（对小团队）**。跨服务排查 Bug 极其困难，链路追踪复杂。     | **极高**。表现层（MVC）负责接口交互，逻辑层（分层）负责核心业务，代码极其清晰。    |
| **可扩展性**     | **弱**。代码高度耦合，加新功能容易牵一发而动全身。                     | **中**。逻辑结构容易扩展，但物理上仍是单体应用。           | **极高**。每个微服务都能独立扩容和升级，适合海量并发。          | **中高**。虽然是单体部署，但由于内部解耦良好，未来随时可以平滑升级为微服务。       |
| **性能**       | **极高**。本地方法直接调用，没有任何中间环节损耗。                     | **高**。本地方法调用，仅有对象在不同层级传递的微小开销。       | **中低**。大量依靠网络请求（HTTP/RPC）通信，有延迟和序列化开销。 | **优**。内部方法级调用，无网络延迟，满足绝大多数常规并发需求。              |
| **团队学习成本**   | **极低**。只需要具备会写基础代码的能力和路由知识。                     | **中等**。需学习面向对象设计原则（如依赖注入）。           | **极高**。需精通 Docker、容器编排、消息队列、分布式事务等中间件。 | **中等偏低**。只需要团队掌握了主流开发框架的“套路”，就能快速进入流水线开发模式。    |
| **10 周内可行性** | **极高**。但期末答辩时，代码可能会没有设计模式和过度臃肿。                 | **高**。能按时做完，且代码规范严谨。                 | **极低**。10 周时间只够配环境，核心业务做不完。            | **适配**。能保证 10 周内按时交付完整业务。                      |
