# AI 协作契约

## 团队名称：Action

## 项目名称：Camera 约拍服务平台

## 日期：2026 年 4 月 2 日

---

## 第一条：AI 使用原则

1. AI 是辅助工具，工程师是责任主体。团队成员必须理解 AI 生成的每一行代码和每一段文字。
2. 所有 AI 输出必须经过人工审查后才能合入项目，禁止"复制粘贴不看"。
3. 使用 AI 的目的是提升效率，而非替代思考。遇到关键设计决策时，应优先团队讨论。

## 第二条：AI 使用范围

### 允许使用 AI 的场景

- 生成代码框架和模板代码（如 CRUD 接口、页面骨架）
- 辅助编写项目文档初稿（需求文档、设计文档等）
- 调试和排查 Bug（分析报错信息、提供修复建议）
- 生成单元测试用例
- 学习新技术时的答疑（如 Vue 3 语法、Spring Boot 配置）
- 代码重构建议

### 禁止直接使用 AI 输出（需额外审查）的场景

- 核心业务逻辑（如订单状态机、信用分计算）
- 数据库设计（表结构、索引策略）
- API 接口设计
- 安全相关代码（登录认证、数据加密、权限控制）

### 完全禁止使用 AI 的场景

- AI 协作反思日志（必须个人独立撰写）
- 用户访谈和问卷调研过程
- 团队内部的架构决策讨论和投票
- 答辩演示内容的核心观点

## 第三条：过程记录要求

1. 所有与 AI 的交互 Prompt 必须保留日志，存放于各阶段的 `docs/PX/prompts/` 目录下
2. Git commit message 必须标注以下前缀之一：
   - `[Human-written]` —— 完全手写
   - `[AI-assisted]` —— AI 辅助生成，经人工审查修改
3. 每阶段按时填写《AI 协作反思日志》

## 第四条：代码合入规则

1. AI 生成的代码必须通过至少一名团队成员的 Code Review，通过 GitHub Pull Request 流程合入
2. 每位成员每个编码阶段（P4、P5）至少有 2 个完全手写的核心函数
3. Pull Request 描述中需注明哪些部分使用了 AI 辅助
4. 发现 AI 生成的代码有安全隐患或逻辑错误时，审查者有权直接拒绝合入

## 第五条：违约处理

1. 首次违约：团队内部提醒，在当周反思日志中记录
2. 再次违约：违约成员需在下一阶段额外完成一份 AI 使用反思报告
3. 严重违约（如未经审查直接合入大量 AI 代码导致项目出问题）：团队讨论后报告指导老师
